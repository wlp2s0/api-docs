    Introduction

    Welcome to the Revolut Open Banking API v3.1! Here you will find all the information to get started with Open Banking API. We're building a future beyond banking, we're excited to have you onboard.

    There is two ways to get access to our Open Banking API:

    For businesses that can do so, we recommend to enroll as a third party provider (TPP) in Open Banking Directory. Follow the Open Banking API v3.1 documentation. We support the following OB certificates: OBWAC (Open Banking QWAC-like certificates), OBSEAL (Open Banking QSEAL-like certificates) and OB non-eIDAS like certificates for transport and signing.
    For TPPs that cannot get registered in the Open Banking Directory and want to integrate with our Open Banking API directly, it is possible to use valid eIDAS QWAC or QSEAL certificates from any of the following QTSPs: Izba Rozliczeniowa S.A., InfoCert S.p.A. If you require a new QTSP to be added please contact us and the QTSP will be added. After having obtained all information, our SLA is to do this within two weeks.

    This Open Banking API can be used to access the accounts of customers of both Revolut Ltd (UK) and Revolut Payments UAB (Lithuania).

    Open Banking Developer Portal

    What is Open Banking Developer Portal?

    If you are building a great app and would like to integrate it with Revolut, then Open Banking Developer Portal is for you. Open Banking Developer Portal allows you to easily manage the applications which are developed for our Open Banking API.

    1. Sign up to Open Banking Developer Portal

    To get an access to Open Banking Developer Portal you need to sign up to developers.revolut.com/portal/signup. After you sign up, please set your Open Banking organisation id in Profile section. This will help us to link your applications with your developer account during dynamic application registration process of Open Banking API.

    2. Get a Sandbox User account

    Once you created your account and set your organization id, you can ask for a test user account for sandbox environment by mailing to api-requests@revolut.com.

    In your request please mention

    • if you are a AISP or PISP
    • if your have registered on Open Banking Directory or which QTSP have registered with
    • with which Organisation ID you have registered on the portal and
    • your contact details.

    3. Create, review, manage and delete your Apps

    First, double check if you are registered on developers.revolut.com/portal/signup and have set your open banking organisation id or another unique identifier in your profile.

    Then you can create Open Banking Applications using the dynamic client registration endpoint.

    Subsequently you can review and delete your Open Banking Applications in the Open Banking Developer Portal. If something has not been setup properly we recommend deleting your app in the Open Banking Developer Portal and recreating it through dynamic registration.

    Open Id Configuration URLs

    • Production: https://oba.revolut.com/openid-configuration
    • Sandbox: https://sandbox-oba.revolut.com/openid-configuration

    Usage and limits

    In order to ensure safe and fair resource sharing, we define the following usage limits:

    • Maximum 16 requests per second for one application

    Please make sure to stay under this limit. Should you estimate this limit be a problem for your usecase of our API, please reach out to us directly.

    Dynamic Client Registration

    In order to create an Application in the Open Banking Developer Portal you will need to use Dynamic Registration.

    The register endpoint is implemented as it is described in the official documentation

    Register client

    curl -X POST https://sandbox-oba.revolut.com/register

    curl -X POST https://oba.revolut.com/register
    Before using this endpoint, first register to Open Banking Developer Portal and set either your `open banking organisation id` when registered on the open banking directory or an alternative `unique identifier` in your profile.
    In order to change your configuration: go to the developers.revolut.com, delete your application and re-register.

    Software statement format

    If a TPP prefers to register directly without using Open Banking Directory, it's possible to use a self-signed software statement. In such case the SSA should satisfy the following format:

    Header

    Header Parameter Value
    alg none

    Body

    Software statement payload

    {
  "org_jwks_endpoint": "https://tpp.com/keys.jwks",
  "org_id": "organisation-id from dev-portal",
  "software_client_name": "app name",
  "software_redirect_uris": ["http://redirect_url"]
}
    Claim Format Description
    org_jwks_endpoint URL a JWKS location
    org_id String an organisation ID from the Open Banking Developer Portal
    software_client_name String a name of a software (shown in the UI)
    software_redirect_uris URL[] a list of allowed redirect URLs

    Signature

    The signature should be an empty string.

    Certificate DN

    Please note that Dynamic Registration requires using the RFC2253 DN format in your request. To obtain the DN of your certificate in this format, you may use the following command: openssl x509 -in /path/to/your/cert.cer -noout -subject -nameopt RFC2253

    This endpoint is for creating payment consent as described in Open Banking API documentation. Domestic Payments are used for local payments in the local currency of the user. Revolut supports only GBP as local in the UK and EUR in SEPA countries.

    Since Revolut Business users can have multiple accounts in the same currency, we allow them to choose which account they want to be charged in the consent authorization UI.

    As part of API requirement you must pass x-jws-signature header in your request. The signature must be generated from request body with TPP signing key that is specified in the JWS header. As specified in the documentation, you must only pass head and signature in <jws_head>..<jws_signature> format.

    Unfortunately, cannot support the EndToEndIdentification field as it is described in the documentation. We will update the documentation once we get support for it.

    Errors

    The API uses the following error codes:

    Code Meaning
    400 Bad Request -- Your request is invalid.
    401 Unauthorized -- Your access token is wrong.
    403 Forbidden -- Access to the requested resource or action is forbidden.
    404 Not Found -- The requested resource could not be found.
    405 Method Not Allowed -- You tried to access an endpoint with an invalid method.
    406 Not Acceptable -- You requested a format that isn't JSON.
    429 Too Many Requests -- You're sending too many requests.
    500 Internal Server Error -- We had a problem with our server. Try again later.
    503 Service Unavailable -- We're temporarily offline for maintenance. Please try again later.
